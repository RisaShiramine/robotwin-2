# DeepSeek task decomposition workflow

This workflow converts RoboTwin task instructions into structured planner supervision with the DeepSeek API and then converts those task-level decompositions into per-timestep planner labels.

If you want a one-command end-to-end pipeline for training data preparation, you can now use:

```bash
bash policy/DP3/process_planner.sh --zarr=/path/to/name.zarr
```

This wrapper will auto-run instruction aggregation, DeepSeek decomposition with resume enabled, planner label building, and final verification. If it cannot infer the instruction directory automatically, pass `--instruction-dir=/path/to/episode_jsons`.

## Step 0: batch-process official RoboTwin instruction files

If you already have the official RoboTwin per-episode instruction files (`episode0.json`, `episode1.json`, ...), first aggregate them with `process_instruction_dir.py`.

```bash
python policy/DP3/scripts/process_instruction_dir.py \
  /path/to/instructions \
  --aggregated-output policy/DP3/examples/official_instruction_batch.json \
  --episode-map-output policy/DP3/examples/official_episode_instruction_map.json \
  --decomposition-input-output policy/DP3/examples/official_decomposition_input.json
```

This produces:

- one aggregated `{seen, unseen}` file for all instructions
- one per-episode instruction map that can be fed directly into `build_planner_labels.py`
- one optional decomposition-input batch file showing the exact instruction subset that will be sent to DeepSeek

By default, batch decomposition uses `--decompose-scope episode_map`, which only sends the chosen per-episode instruction set to DeepSeek instead of every paraphrase in every file. This keeps official RoboTwin instruction folders tractable when episodes 0-49 contain many near-duplicate instructions. Use `--decompose-scope all` only if you explicitly want every aggregated paraphrase.

If you also want to call DeepSeek in the same step, add `--decompose` plus decomposition output paths.

## Step 1: generate task decompositions

### Input format

The CLI expects a JSON file with two top-level lists:

```json
{
  "seen": ["instruction 1", "instruction 2"],
  "unseen": ["instruction 3"]
}
```

### Environment

Set a DeepSeek API key before running the script:

```bash
export DEEPSEEK_API_KEY=your_key_here
```

### Run decomposition

```bash
python policy/DP3/scripts/decompose_tasks.py \
  policy/DP3/examples/task_decomposition_input.json \
  --output policy/DP3/examples/task_decomposition_output.json \
  --jsonl-output policy/DP3/examples/task_decomposition_output.jsonl \
  --deduplicate
```

For unstable networks, it is recommended to add:

```bash
--resume --timeout 180 --max-retries 8
```

`--resume` reuses rows already present in the existing output files and `decompose_tasks.py` now flushes progress after each successfully decomposed instruction, so an interrupted run can continue instead of restarting from scratch.

### Decomposition output shape

Each instruction is expanded into a normalized planner record with:

- `instruction`
- `canonical_task`
- `task_id`
- `task_category`
- `scene_objects`
- `final_goal`
- `stages[]`

Each `stages[]` entry contains:

- `stage`
- `stage_type`
- `action_type`
- `source_object`
- `target_object`
- `support_object`
- `target_support`
- `target_region`
- `target_location`
- `spatial_relation`
- `preferred_arm`
- `required_objects`
- `completed_subgoals_before_stage`
- `success_criteria`

## Step 2: build per-timestep planner labels

Use `build_planner_labels.py` to turn the decomposition output into `planner_labels.jsonl`.

### Required inputs

You need:

1. A decomposition file generated by `decompose_tasks.py` (`.json` or `.jsonl`)
2. A mapping from `episode_id` to the exact instruction used in that episode
3. Episode transition lengths (either from a JSON file or directly from a DP3 zarr via `--dp3-zarr`)

The safest setup is to pass the exact DP3 zarr that training will read. In that mode, the builder derives episode lengths from `meta/episode_ends`, so the `flat_index`, `episode_id`, and `timestep` layout comes from the same source of truth as the training dataset.
### Minimal example

```bash
python policy/DP3/scripts/build_planner_labels.py \
  --decomposition-file policy/DP3/examples/task_decomposition_output.json \
  --episode-instruction-map policy/DP3/examples/episode_instruction_map.json \
  --episode-lengths-json policy/DP3/examples/episode_lengths.json \
  --output policy/DP3/examples/planner_labels.jsonl \
  --summary-output policy/DP3/examples/planner_labels_summary.json
```

If your `episode_lengths.json` is incomplete, or you do not want to maintain it manually, pass the DP3 zarr instead:

```bash
python policy/DP3/scripts/build_planner_labels.py \
  --decomposition-file policy/DP3/examples/official_task_decomposition_output.json \
  --episode-instruction-map policy/DP3/examples/official_episode_instruction_map.json \
  --dp3-zarr policy/DP3/data/your_task.zarr \
  --output policy/DP3/examples/planner_labels.jsonl \
  --summary-output policy/DP3/examples/planner_labels_summary.json
```

### Alternative input mode

If you already have per-episode instruction files like `episode0.json`, `episode1.json`, ... you can use:

```bash
python policy/DP3/scripts/build_planner_labels.py \
  --decomposition-file policy/DP3/examples/task_decomposition_output.json \
  --instruction-dir /path/to/instructions \
  --instruction-source seen_first \
  --episode-lengths-json policy/DP3/examples/episode_lengths.json \
  --output policy/DP3/examples/planner_labels.jsonl
```

### Alignment mode

By default the script uses **zarr velocity boundary inference**: it looks at the DP3 zarr `state` / `action` pairs, computes a simple motion proxy `||action - state||`, detects low-velocity plateaus, and uses those near-stop moments as stage boundaries. This is meant to better match RoboTwin execution videos than a pure uniform split.

By default the script also applies **stack-stage normalization** before alignment. For stack tasks, the decomposition is expanded into an execution-oriented per-block sequence following the color/object order in the instruction:

- `align_*`
- `grasp_*`
- `transport_*`
- `place_release_*`
- `retreat_reset_*`

If you want to fall back to the previous even split, add:

```bash
--boundary-mode uniform
```

If you want to keep the raw decomposition stages, add:

```bash
--disable-stage-normalization
```

If you already have explicit stage boundaries, pass a JSON file like:

```json
{
  "0": [
    {"start": 0, "end": 2},
    {"start": 2, "end": 4},
    {"start": 4, "end": 6}
  ]
}
```

with:

```bash
--stage-boundaries-json your_boundaries.json
```

### Planner label output shape

Each output JSONL row contains:

- `flat_index`
- `episode_id`
- `timestep`
- `instruction`
- `split`
- `canonical_task`
- `task_category`
- `final_goal`
- `stage`
- `stage_type`
- `stage_index`
- `num_stages`
- `is_terminal_stage`
- `source_object`
- `target_object`
- `support_object`
- `target_region`
- `target_support`
- `target_location`
- `spatial_relation`
- `preferred_arm`
- `required_objects`
- `completed_subgoals`
- `relevance_objects`
- `success_criteria`

This output is meant for planner supervision. DP3 should continue reading the original zarr with `point_cloud`, `state`, and `action`.

## Step 2.5: verify that episodes are correct and aligned with the zarr

If you are asking “how do I know the decomposed episode is correct and that it lines up with the zarr training data?”, there are two separate checks:

1. **Semantic correctness**: inspect whether the chosen instruction for each `episode_id` maps to the intended decomposition.
2. **Index alignment correctness**: verify that `planner_labels.jsonl` has the exact same transition layout as the DP3 zarr.

For semantic correctness, first inspect the builder summary JSON:

- `missing_instruction_matches` tells you which episodes could not be matched to a decomposition at all.
- `matched_episode_ids` lists the episodes that were actually written.
- `validation.per_episode_lengths` shows how many planner rows were emitted per episode.

For index alignment correctness, run the verifier against the same zarr you will train on:

```bash
python policy/DP3/scripts/verify_planner_labels.py \
  --planner-labels policy/DP3/examples/planner_labels.jsonl \
  --dp3-zarr policy/DP3/data/your_task.zarr \
  --episode-instruction-map policy/DP3/examples/official_episode_instruction_map.json \
  --summary-output policy/DP3/examples/planner_labels_verify_summary.json
```

This verifier will fail fast if any of the following are wrong:

- `planner_labels.jsonl` row count does not equal the zarr transition count
- `flat_index` is not contiguous from `0..N-1`
- any row's `episode_id` disagrees with `meta/episode_ends`
- any row's `timestep` disagrees with its within-episode offset
- the instruction stored in `planner_labels.jsonl` does not match the instruction map for that episode

At training time, `PlannerRobotDataset` now repeats the same structural checks when it loads `planner_labels.jsonl`, so a misaligned file will raise an error before optimization starts.

A practical workflow is:

1. Build labels with `--dp3-zarr` whenever possible.
2. Open the summary JSON and confirm `missing_instruction_matches` is empty.
3. Run `verify_planner_labels.py` and confirm `"passed": true`.
4. Only then point training at that zarr + planner label pair.

## Step 3: use planner labels for training

A first training hook is now available in `diffusion_policy_3d/dataset/planner_robot_dataset.py`.
It loads the original DP3 zarr plus `planner_labels.jsonl`, aligns them by `flat_index`, and returns a `planner` dict containing integer ids for stage and object supervision.

This is intended as the data interface for planner-conditioned training; model and loss integration are still up to the training code.
